# v2.5 — Bâtir le surrogate mouillé/sec & juger (le geste qui sert le but)

## Le recadrage (pourquoi ce spec, et pas un oracle C-G)

L'étape précédente envisagée — bâtir un oracle Carrier-Greenspan pour mesurer le dernier
axe (étendue) au **grade analytique** — était un tapis roulant, par les deux signes :

1. **Critère mesuré ≠ critère de succès.** Le livrable se juge sur la **plausibilité
   visuelle du rendu**, pas sur une L2 contre un oracle analytique. C-G valide au grade
   analytique, au-delà du but.
2. **La prochaine mesure ne change plus la décision.** L'axe étendue (la seule chose qui
   pourrait encore basculer vers l'encodeur) **tombe gratuitement** en bâtissant le
   surrogate : le **k combiné du vocabulaire réel EST l'endpoint**. Un seul POD ; la
   courbe de saturation n'est que de l'explication.

Donc : **bâtir le surrogate**. Les deux verdicts (étendue + opérateur) en sortent.
**Carrier-Greenspan / Synolakis** est démoté de « prérequis » à **outil de débogage
contingent** : sorti *seulement si* le run-up d'île rendu paraît faux, pour distinguer
bug-solveur de limite-de-base. Le run-up est déjà couvert au grade visuel par **Thacker**
(trait d'eau mobile sur paroi parabolique = montée/descente sur pente).

## Goal

Bâtir le surrogate mouillé/sec — base POD + **un seul opérateur linéaire global** (DMD,
hypothèse single-global de v2, maintenant en mouillé/sec) — sur le **vocabulaire borné
réel** ; lire le **k combiné** (endpoint d'étendue) et **juger les rollouts rendus à
l'œil** (plausibilité visuelle = critère de succès).

## Composants

### 1. Génération du vocabulaire (réutilise `terrains.py`, `solver_wetdry.py` 2e ordre)
~16 scénarios, par le solveur 2e ordre **validé**, **pesés vers le run-up** :
- **Terrains** : plat, pente (plage), île (gaussienne). [réutiliser/étendre `terrains.py`]
- **Fronts** : dam-break sec (mouillage), bore mouillé (Stoker), **run-up d'onde solitaire
  (sur-représenté)**. Le run-up est le seul à exercer l'**assèchement** (h>0→h=0, run-down)
  ET le mouillage — le cas le plus dur/risqué de l'opérateur. Dam-break-lourd sous-testerait
  cet axe → **plusieurs run-ups** à travers terrains × orientations.
- **Positions × orientations** : positions de front/d'île **étalées** (pas seulement les
  orientations) ; orientations {x, y, diagonale}. **Amplitude de transport de l'échantillon
  ≥ celle du livrable** → saturation sur l'échantillon (plus large) ⇒ vocabulaire réel
  (plus étroit) borné *a fortiori* (verdict conservateur).
- IC onde solitaire : `η = H·sech²(γ(x−x₀))`, `γ=√(3H/(4d³))`, sur la plage. [helper à créer]

### 2. POD combiné → endpoint d'étendue (réutilise `pod.py`)
`fit_pod` sur **tous** les snapshots empilés en un jeu → **k combiné**.
- **Verdict étendue** : k **sature** (~50, sous-linéaire en N de scénarios) → étendue
  **mesurée-bornée** → base linéaire suffit. k **croît ~linéairement** (~16×24≈384,
  recouvrement nul) → étendue non bornée → **encodeur justifié**. (Courbe nested
  optionnelle ; l'endpoint décide.)

### 3. Opérateur global unique (réutilise `dmd.py`)
Ajuster **un seul** opérateur linéaire sur les coefficients POD combinés → rollout → décode.
Teste l'hypothèse **single-global de v2** en mouillé/sec : une carte linéaire avale-t-elle
dam-break + run-up + 3 terrains ? Le spread du vocabulaire est ce qui exerce ce test.

### 4. Rendu & jugement (réutilise `render.py`)
Rollout surrogate vs vérité solveur, côte à côte, pour scénarios représentatifs (run-up
inclus). **La plausibilité visuelle est le critère de succès — jugée par l'utilisateur.**

## Les deux verdicts (tombent du build)
- **Étendue / base** : k combiné (sature vs croît).
- **Opérateur / visuel** : un seul opérateur global roule-t-il tous les régimes
  plausiblement ? (jugé à l'œil)

Si les deux tiennent → le surrogate linéaire suffit pour le vocabulaire borné, **mesuré
sur chaque axe + jugé au critère du but**. Si la base explose → encodeur. Si l'opérateur
casse visuellement → modèle plus riche côté dynamique (pas forcément encodeur de base).

## Hors-scope
- Oracle C-G/Synolakis analytique (contingent débogage seulement).
- Courbe de saturation nested (explication ; l'endpoint décide).
- Fidélité L2 du champ de vitesse (hors critère visuel).

## Contraintes
- Solveur 2e ordre **validé** comme générateur (ne pas re-valider) ; 1er ordre / `solver.py`
  intacts. Réutiliser `pod.py`, `dmd.py`, `render.py`, `terrains.py`. numpy/scipy/matplotlib.
- Outputs/ gitignoré ; docs committés.

## Acceptation
Le build produit : (a) le **k combiné** rapporté (endpoint d'étendue, avec son seuil
sature/croît) ; (b) des **rendus rollout vs vérité** pour ≥3 scénarios dont ≥1 run-up,
prêts au jugement visuel. Pas de critère L2 d'acceptation (le critère est visuel).
